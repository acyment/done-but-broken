# CP13: Global audit feed for the customer portal

The portal needs one chronological activity stream across all aggregates. Maintain a global audit feed: every applied event appends one feed entry carrying feed_seq (global, starting at 1, gap-free in application order), the aggregate_id, and the same per-aggregate seq, event_id, event_type, and at as the aggregate's audit entry. The audit_feed query returns feed entries and supports after_feed_seq (strictly greater), event_types (filter applies before the limit), and limit (positive integer). Frozen behavior: per-aggregate audit sequence numbers are NEVER renumbered from the global counter — auditLog output for every existing scenario must stay byte-identical. The feed is derived presentation state: it is EXCLUDED from the canonical state registry (like the applied-event bookkeeping), so no registry delta and every earlier worked hash is unchanged.

## Invariants introduced (cumulative from now on)

- I-SEQ re-asserted across the feed addition: per-aggregate sequence numbers stay gap-free and unchanged.

## Scenarios

#### Scenario: Feed spans aggregates
- **GIVEN** events on a subscription and its invoice
- **WHEN** audit_feed is queried
- **THEN** entries appear in application order with global feed_seq 1..n and each entry's per-aggregate seq unchanged

#### Scenario: Pagination
- **GIVEN** a feed with three entries and after_feed_seq=1 limit=1
- **WHEN** audit_feed is queried
- **THEN** exactly the second entry is returned

#### Scenario: Type filter before limit
- **GIVEN** a feed with mixed event types
- **WHEN** audit_feed is queried with event_types
- **THEN** only matching entries count toward the limit

## Queries introduced

- `audit_feed` with fields: `after_feed_seq`, `event_types`, `kind`, `limit`
