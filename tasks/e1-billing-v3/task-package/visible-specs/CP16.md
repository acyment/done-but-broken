# CP16: Finalized invoices are immutable

invoice_finalized freezes an invoice. invoice_recomputed reprices an open invoice in place; recomputing a finalized invoice instead writes a new invoice document at the id given by new_invoice_id (the convention is the original id with suffix -r1, -r2, ...) and leaves the original byte-for-byte untouched. Registry delta: append `finalized` to the invoice registry list (after `refunded_cents`); as a boolean it is omitted while false.

## Invariants introduced (cumulative from now on)

- I-IMMUT: no event mutates a finalized invoice; recomputation produces new documents.

## Scenarios

#### Scenario: Open recompute is in place
- **GIVEN** an open invoice and an invoice_recomputed event
- **WHEN** the invoice view is queried
- **THEN** the same invoice id carries the repriced lines

#### Scenario: Finalized recompute forks
- **GIVEN** a finalized invoice and an invoice_recomputed event
- **WHEN** both invoice views are queried
- **THEN** the original is byte-for-byte unchanged and a new -r1 document carries the repriced lines

## Events introduced

- `invoice_finalized` with fields: `at`, `event_id`, `invoice_id`, `type`
- `invoice_recomputed` with fields: `at`, `event_id`, `invoice_id`, `new_invoice_id`, `type`
