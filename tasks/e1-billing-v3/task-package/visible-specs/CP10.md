# CP10: Partial refunds with a hard cap

payment_refunded events refund part of a captured invoice. Refunds accumulate; a refund that would push the refunded total past the captured amount is a complete no-op (no partial application, no audit entry). Registry delta: append `refunded_cents` to the invoice registry list (after `discount_total_cents`) and append `refunded_cents` to the line registry list (after `discount_cents`).

## Invariants introduced (cumulative from now on)

- I-REFCAP: refunded total never exceeds the captured amount per invoice.

## Scenarios

#### Scenario: Partial refund accumulates
- **GIVEN** a captured invoice and two partial refunds within the cap
- **WHEN** the invoice view is queried
- **THEN** the refunded amount is their sum

#### Scenario: Over-cap refund is a no-op
- **GIVEN** a refund exceeding the remaining refundable amount
- **WHEN** the invoice view and audit log are queried
- **THEN** the refunded amount is unchanged and no audit entry was added

## Events introduced

- `payment_refunded` with fields: `amount_cents`, `at`, `event_id`, `invoice_id`, `type`

## Worked examples

Each example is `evaluate(events, query) -> expected`, verbatim. These exact cases are the checkpoint's acceptance examples.

        "at": "2026-01-01T00:00:00.000Z",
        "subscription_id": "S1",
        "customer_id": "C-S1",
        "plan_id": "basic",
        "plan_price_cents": 3000,
        "period_start": "2026-01-01",
        "period_end": "2026-01-31"
      },
      {
        "event_id": "INV-1-gen",
        "type": "invoice_generated",

      {

      },
      {
        "event_id": "ref-1",

  }
}
