# CP03: Idempotent payment capture

Record payment captures against invoices. Webhooks retry, so the same capture event can arrive more than once.

## Invariants introduced (cumulative from now on)

- I-IDEM: a payment_captured event whose event_id was already applied is a complete no-op: no state change and no audit entry.

## Scenarios

#### Scenario: Capture marks the invoice paid
- **GIVEN** a generated invoice and a payment_captured event for its full total
- **WHEN** the invoice view is queried
- **THEN** the captured amount equals the total

#### Scenario: Duplicate capture is a no-op
- **GIVEN** the same payment_captured event applied twice
- **WHEN** the invoice view and audit log are queried
- **THEN** the captured amount counts once and the audit log has no duplicate entry

## Events introduced

- `payment_captured` with fields: `amount_cents`, `at`, `event_id`, `invoice_id`, `type`

## Worked examples

Each example is `evaluate(events, query) -> expected`, verbatim. These exact cases are the checkpoint's acceptance examples.

}
