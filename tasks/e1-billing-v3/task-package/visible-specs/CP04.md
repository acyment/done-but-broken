# CP04: v1 invoice serialization (legacy byte stability)

Expose the legacy v1 invoice serializer used by downstream partners. The v1 format has a fixed legacy field order and omits absent optional fields entirely; partners parse it positionally, so its bytes must never change for records it can already serialize.

## Invariants introduced (cumulative from now on)

- I-V1BYTES: serialize_v1 output stays byte-identical for previously serializable invoices, forever.

## Scenarios

#### Scenario: v1 serialization is stable
- **GIVEN** a paid invoice
- **WHEN** serialize_v1 is queried
- **THEN** the output is the exact documented byte string with the legacy field order

## Queries introduced

- `serialize_v1` with fields: `invoice_id`, `kind`

## Worked examples

Each example is `evaluate(events, query) -> expected`, verbatim. These exact cases are the checkpoint's acceptance examples.
