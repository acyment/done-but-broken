# CP01: Subscription creation and the audit trail

The billing engine is event-sourced: `applyEvent(state, event)` folds events into an immutable state value, and `evaluate(events, query)` answers one query over a folded event list. Create subscriptions and keep a complete audit trail.

## Invariants introduced (cumulative from now on)

- I-STATE: subscriptions move only along the documented lifecycle (trialing -> active -> past_due -> canceled).
- I-SEQ: every applied event appends exactly one audit entry to its aggregate, with sequence numbers starting at 1 and gap-free per aggregate.

## Scenarios

#### Scenario: Active subscription created
- **GIVEN** a subscription_created event without the trial flag
- **WHEN** the subscription view is queried
- **THEN** the status is active with the plan and period from the event

#### Scenario: Trial subscription activated
- **GIVEN** a subscription created with trial=true and a subscription_activated event
- **WHEN** the subscription view is queried
- **THEN** the status is active

#### Scenario: Audit sequence is gap-free
- **GIVEN** a subscription with two applied events
- **WHEN** the audit log for the subscription is queried
- **THEN** the entries carry sequence numbers 1 and 2 in application order

## Events introduced

- `subscription_activated` with fields: `at`, `event_id`, `subscription_id`, `type`
- `subscription_created` with fields: `at`, `customer_id`, `event_id`, `period_end`, `period_start`, `plan_id`, `plan_price_cents`, `subscription_id`, `trial`, `type`

## Queries introduced

- `audit_log` with fields: `aggregate_id`, `kind`
- `subscription` with fields: `kind`, `subscription_id`
