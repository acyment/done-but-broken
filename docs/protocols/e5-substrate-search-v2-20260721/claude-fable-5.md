# Choosing a Substrate for a Spec-Execution Experiment on AI Coding Agents

## TL;DR
- **The substrate you want barely exists, and your own screen won't catch the deeper trap.** Densely dependent, public-surface change chains are genuinely rare in mature repos — every benchmark program that claims them (ChainSWE, SWE-Bench-CL, EvoClaw) admits its "dependency" labels are file/AST co-location, exactly your Lesson 1. The one candidate I can fully specify with named per-edge mechanisms is the **FastAPI streaming-response chain (PRs #14987 → #15022 → #15030 → #15038 → #15588, merged Feb–Apr 2026)**.
- **Your four lessons all guard against false-positive dependencies; none guards against non-discrimination.** A clean-traceback dependency (your verified-real edge shape) is precisely the kind BOTH arms catch via shell + public tests — so it won't separate spec-runner from spec-reader. The edges that discriminate are *silent* ones, which your subtract-one-change check does not distinguish from loud ones.
- **Recommendation: run one instrumented pilot on FastAPI for a *scoped* claim** ("executable acceptance specs reduce false-done declarations when a later change consumes an API surface an earlier change introduced"), not a general benchmark. Budget and solo-operator constraints make a general claim unsupportable, and honesty on that point is more LinkedIn-proof than five shaky nominations.

## 1. Critique of the Criteria (read this first)

Your four lessons form an excellent filter against *fabricated* dependency — they kill co-location masquerading as causation, curriculum-reordering artifacts, merge-commit-blocked replay, and tests that don't touch the claimed edge. But every one of them protects the same thing: that the dependency **exists and is attributable**. None of them protects the property your *experiment* actually needs, which is that the dependency is **discriminating** for a two-arm spec-vs-no-spec design. Your own verified-real edge — "a function signature change that breaks downstream callers with a clean traceback" — is the clearest illustration of the problem: a loud, attributable break is exactly what an agent in *either* arm detects through the public test suite and normal shell access. The acceptance-spec-runner arm gains no advantage on a failure that `pytest` and an import error already surface. The edges that would actually separate the two arms are the *silent* ones: where removing the earlier change leaves the later change importing, compiling, and passing its public tests, but changing a behavior that only the held-out acceptance scenario exercises. Subtract-one-change tells you the edge is real; it does not tell you the edge is silent, and silence is the whole game. You are, in effect, still screening for the wrong kind of real dependency.

The second blind spot is that you are screening the *substrate* while the threat to a public-attack-proof claim lives in the *design*. Three specific failure modes your lessons don't touch: (a) **state-accumulation confound** — carrying one workspace across 4–8 dependent changes means an early mistake by the agent poisons every later task. ChainSWE (Qirui Jin et al., arXiv 2607.02606), which mines chronological chains of 304 issues across 54 Python projects from six SWE-bench-family datasets, reports "per-bug success dropping by up to 70% at the deepest chain positions," and isolates the cause with an "Oracle" per-bug baseline whose gap to the sequential runs "quantifies how much performance is lost due to accumulated agent and repo state rather than genuine bug difficulty." Your two arms may differ not because one had an executable spec but because one arm happened to corrupt the workspace earlier — you must measure and control for this. (b) **Contamination beyond the diff** — your proposed "reproduce the diff from the issue" check catches memorized *fixes*, but frontier models memorize the *library's public API shape* from documentation regardless of commit date. "The SWE-Bench Illusion" (Liang et al., arXiv 2506.12286) found SoTA models reach "up to 76% accuracy in identifying buggy file paths using only issue descriptions, without access to repository structure" (OpenAI's o3 at 76%), collapsing to "up to 53% on tasks from repositories not included in SWE-Bench" — direct evidence of memorization that a post-January-2026 commit date alone does not neutralize if the pattern is long-documented. (c) **Audience-representativeness self-own** — the "public API/CLI/HTTP boundary + mature Cucumber runner" requirement biases you hard toward web frameworks, which is the exact "that looks nothing like my codebase" objection you fear, aimed at a CTO reading on their phone. Decide now whether an HTTP-framework result generalizes in your narrative, or the substrate wins the methods fight and loses the comment section.

## 2. Nominations

I am deliberately nominating **one fully-specified candidate and two partially-specified ones**, and telling you plainly which edges I expect to fail your check. Quantity here would be dishonest.

### Candidate A — FastAPI streaming responses (introduce-then-use API chain) — STRONGEST
- **Repository:** `github.com/fastapi/fastapi`
- **Base commit:** `master` immediately before PR #14987 (FastAPI ~0.132.x, ~Feb 23 2026).
- **Changes (chronological, each a squash-merged single commit on `master`):**
  1. **PR #14987** "Add support for Starlette 1.0.0+" — merged Feb 24 2026 (squash, commit `c73bc94`), shipped 0.133.0.
  2. **PR #15022** "Add support for streaming JSON Lines and binary data with `yield`" — merged Feb 27 2026 (commit `749cefd`), shipped 0.134.0. Introduced generator detection (`Dependant.is_gen_callable`/`is_async_gen_callable`), `get_stream_item_type` (fastapi/dependencies/utils.py), and `stream_item_field` used for validation + OpenAPI + Rust serialization (fastapi/routing.py). Carried the Starlette floor bump: per the PR body and 0.134.0 release notes (released 27 Feb 2026, 21:17), "This also upgrades Starlette from >=0.40.0 to >=0.46.0, as it's needed to properly unrwap and re-raise exceptions from exception groups." Tests arrived with it (JSON Lines, streaming-with-yield, PNG example, cancellation).
  3. **PR #15030** "Add support for Server Sent Events" — merged Mar 1 2026 (commit `2238155`), shipped 0.135.0. Added `fastapi/sse.py` (`EventSourceResponse`, `ServerSentEvent`); 23 files changed; added `tests/test_sse.py` + `tests/test_tutorial/test_server_sent_events/test_tutorial001–005.py`.
  4. **PR #15038** "avoid yield from a TaskGroup, only as an async context manager, closed in the request async exit stack" — shipped 0.135.1 (Mar 1 2026). Fixes the SSE producer/consumer task-group path.
  5. **PR #15588** "Validate Server Sent Event fields to avoid applications from sending broken data" — later SSE hardening on the `ServerSentEvent` class.
- **Per-consecutive-pair dependency mechanisms:**
  - **#15022 → #15030 (the real introduce-then-use edge):** SSE reuses the generator-detection + `get_stream_item_type` + `stream_item_field` machinery that #15022 introduced. FastAPI's own docs (marked "Added in 0.134.0"/"0.135.0") make the shared path explicit: "If you declare the return type as AsyncIterable[Item], FastAPI will use it to validate, document, and serialize the data using Pydantic. As Pydantic will serialize it in the Rust side, you will get much higher performance." An SSE route declared `-> AsyncIterable[Item]` runs through that same typed-serialization path. Remove #15022 and #15030's typed-SSE tests should fail at the type-extraction/serialization layer.
  - **#15030 → #15038:** #15038 patches the TaskGroup/async-exit-stack handling *introduced by* #15030's SSE architecture. Remove #15030 and there is no code path for #15038 to touch — its regression test has nothing to bind to.
  - **#15030 → #15588:** #15588 validates fields of the `ServerSentEvent` class defined in #15030. Remove #15030 and the symbol under test does not exist (loud import error — note: this is a *loud* edge, see the caveat below).
  - **#14987 → #15022:** weakest edge; see failure prediction.
- **History shape:** Linear. FastAPI uses squash-merge (PR #14987 explicitly shows "enabled auto-merge (squash)"; every PR lands as one commit titled "…(#NNNNN)" with a single parent). No merge commits inside the span, so subtract-one-change is structurally clean — revert the target PR's squashed commit, re-run the later PR's tests.
- **Tests:** Arrive with each change, separable into visible/held-out at file granularity (e.g., hold out `tests/test_sse.py`, expose `tests/test_tutorial/test_server_sent_events/*`).
- **Surface:** Pure HTTP (`text/event-stream`, `application/jsonl`), drivable by `httpx`/Starlette `TestClient` → trivially behave/pytest-bdd-scenarioable.
- **Containerization:** `pip install`, no external services, fast suite. Cheap.
- **Leadership story:** "We asked an agent to add JSON-Lines streaming, then Server-Sent Events on top of it, then the cancellation fix — the same sequence FastAPI's maintainer actually shipped in Feb–Mar 2026 — and measured whether it silently regressed streaming when it couldn't run the acceptance spec." Concrete, current, recognizable.
- **Single most likely way it fails your check:** the **#14987 → #15022** edge. #15022 carries its *own* Starlette `>=0.46.0` floor, so subtracting #14987 (broad Starlette 1.0.0+ compat) very likely leaves #15022's tests green — this edge is co-location (both are "Starlette-adjacent work merged the same week"), not dependency. Drop it and start the chain at #15022. Secondary risk: the #15030→#15588 and #15030→#15038 edges are *loud* (missing symbol / stack trace), so while they pass subtract-one-change, they are weak *discriminators* for your two-arm design — the spec-reader arm catches them too.

### Candidate B — Django ORM "fetch modes" cluster (introduce-then-use + a real deprecation edge) — MEDIUM, NEEDS YOUR VERIFICATION
- **Repository:** `github.com/django/django`
- **Anchor:** ticket #28586, PR **#17554** "Added model field fetch modes" (adamchainz; merged into `main` for the 6.1 line), which introduces `QuerySet.fetch_mode()`, the `FETCH_ONE`/`FETCH_PEERS`/`RAISE` constants, and `FieldFetchBlocked`; and in the same release deprecates calling `select_related()` with no arguments ("use the `FETCH_PEERS` fetch mode").
- **Shape / mechanisms:** introduce (`FETCH_PEERS`) → consume (peer-prefetch behavior for deferred fields and related objects) → deprecate-old-path (`select_related()` no-arg). The introduce-then-deprecate pair is a genuine dependency: the deprecation message and the replacement API must both exist.
- **History shape:** Django forbids merge commits on `main` (contributing guide mandates rebase), so the main-branch history is linear — subtract-one-change is structurally feasible. **You must verify the exact commit sequence**; #17554 landed via many force-pushes, so the clean per-commit boundaries you need may have to be reconstructed from `main` commits, not the PR branch.
- **Surface:** ORM is a public Python API (behave/pytest-bdd can drive it directly); Django's test `Client` gives an HTTP surface if you wrap views around it.
- **Single most likely way it fails your check:** the introduce→consume edge may be *loud* (removing `fetch_mode()` breaks `AttributeError` immediately), and the discriminating behavioral edge — "does FETCH_PEERS actually collapse N+1 to 2 queries" — is asserted with `assertNumQueries`, which is a *query-count* oracle, not a functional-regression oracle; an agent can satisfy functional tests while regressing the query-count behavior, which is either exactly the silent edge you want or an artifact you'll reject. Verify with subtract-one-change whether the query-count test fails cleanly with an attributable cause.

### Candidate C — Django database-level `on_delete` options (schema/introduce-then-use) — MEDIUM, PARTIALLY DISQUALIFIED
- **Repository:** `github.com/django/django`
- **Anchor:** ticket #21961, PR **#19925** "Added support for database-level delete options for ForeignKey" (felixxm; merged Oct 18 2025, commit `0c487aa`), introducing `DB_CASCADE`/`DB_SET_NULL`/`DB_SET_DEFAULT` via SQL `ON DELETE` clauses, with an explicitly-discussed follow-up on introspection/`inspectdb` and PR #18116 touching FK-constraint recreation.
- **Shape / mechanisms:** introduce DB-level delete constants + schema-generation path → later changes consume that schema path (introspection, constraint handling). step-N-reads-step-N-1 flavor.
- **Single most likely way it fails your check — and a HARD-EXCLUSION risk:** `DB_CASCADE`/`DB_SET_*` are **only implemented on PostgreSQL and MySQL, not SQLite** (per the merged release notes). That means the discriminating tests require a real database engine, which pushes against your "no external services that cannot run locally" exclusion (a Postgres container is *local* but heavier, and raises solo-operator cost). The follow-up edges are also loosely coupled — I am least sure the introspection follow-up genuinely *requires* #19925's exact code rather than merely touching the same file. This is the candidate most likely to collapse into co-location on verification.

## 5. The Supply Question — your rarity thesis is correct

**My search supports the claim: densely dependent, public-surface change chains are rare in mature repositories, and most consecutive changes are independent.** The strongest evidence is that every serious attempt to build this dataset had to *define dependency down* to co-location — the exact failure you already found:
- **ChainSWE** joins issues into chains when "the latter's gold patch and test patch apply cleanly on top of the former's resolved state" and filters by AST "code-overlap" of modified files/functions/classes, discarding any window "whose bugs have zero pairwise overlap" — i.e., overlap, not semantic API consumption.
- **SWE-Bench-CL** (arXiv 2507.00014, §9 Limitations) states outright that "Dependency awareness relies on overlapping file paths, potentially missing deeper semantic or API-level dependencies," and notes its scope is only "8 open-source Python repositories, each with at most 50 issues per sequence" (273 tasks total).
- **EvoClaw** admits its pipeline "retains only commits that touch source code with non-trivial inter-commit dependencies … it may bias the resulting benchmark toward dependency-rich evolution and underrepresent independent maintenance work" — a direct confession that they had to *filter hard* to find dependency, which only makes sense if it is rare in the wild.

**Where dense chains actually concentrate:**
- **Project type/lifecycle:** fast-moving libraries *adding capability* in a pre-1.0 or major-version run-up (FastAPI's streaming line, pydantic v2.x, uv, polars), **not** mature stable cores (CPython stdlib, NumPy) where the modal commit is an independent bug fix.
- **Subsystem:** serialization/response handling, routing/dispatch, query compilation, and migration/format frameworks — anywhere a *typed or structured surface is introduced and then consumed*. FastAPI streaming sits exactly here.
- **Shape:** introduce-then-use on a public API is the richest and most replayable vein; stacked-PR chains are richest in principle but hardest to find as public artifacts; migrations are clean but sparse; deprecation cycles are real but release-separated and thus unreplayable in one workspace.

**The strongest narrower claim a realistic candidate can support** is not "an executable spec prevents regressions across dependent changes" but: **"When a later change consumes a public API surface that an earlier change in the same session introduced, giving the agent an *executable* acceptance spec (vs. a readable-only one) reduces the rate of false 'done' declarations against held-out behavior."** Scope it to introduce-then-use on a public surface, name the mechanism, and it survives a hostile read.

## 6. What Would Make Me Walk Away

**Substrate classes I would refuse even if they scored well:**
1. **Any deprecation cycle that spans releases** — you cannot honestly replay warn→enforce→remove in one 4–8-change workspace; the dependency is real but the nodes are release-separated. Refuse regardless of how "structural" the dependency looks.
2. **Any edge whose only signal is a loud failure** (missing symbol, import error, clean traceback) as the *primary* discriminating edge — it is caught by both arms and cannot separate spec-runner from spec-reader. Your Lesson-1 "verified real" edge is this class; it validates your *pipeline* but is near-useless for your *experiment*. Use loud edges only as chain scaffolding, never as the measured edge.
3. **Substrates whose discriminating behavior needs a real external engine** (Candidate C's Postgres/MySQL `DB_CASCADE`) — even a local container raises solo-operator cost and invites the "needs infra" objection.
4. Everything already in your hard-exclusion list (reordered curated datasets, metadata-only dependency claims, synthetic projects, behavior-invalidating sequences, private-detail-only assertions).

**And the blunt bottom line:** at low-tens-of-dollars, one operator, no deadline, **no candidate I found can support the general claim** ("executable specs prevent regressions across dependent changes") to a standard that survives public attack, because the general claim requires a *population* of silent, discriminating, dependent edges that the supply data says does not exist in harvestable quantity. What you *can* support is a single, well-instrumented FastAPI pilot for the scoped introduce-then-use claim in §5. If, when you run subtract-one-change on FastAPI, the #15022→#15030 edge turns out to be *silent* (SSE typed-serialization tests fail without #15022 but without a loud import error), you have your one genuinely discriminating edge and a publishable narrow result. If that edge is loud too, walk away from the two-arm framing and publish the supply finding itself — "we went looking for dependent public-surface chains to test spec execution, and the interesting result is how few exist" — which, for a CTO/staff-engineer audience, is the more defensible and more original post.

**Staged next steps and the thresholds that change them:**
1. **Verify FastAPI first (a few dollars of compute).** Revert #15022's squashed commit, re-run #15030's typed-SSE tests. **If they fail without a loud import/attribute error → you have your discriminating edge; promote to a two-arm study.** If they fail loudly → this edge is scaffolding only.
2. **Confirm the shared-mechanism claim by hand** — this is the single most important unverified item (see below). Read `fastapi/sse.py` and `fastapi/routing.py` to confirm the SSE path calls the `get_stream_item_type`/`stream_item_field` machinery, not a parallel copy.
3. **Instrument the state-accumulation confound** — log workspace diffs per step and compare a "reset each step" oracle arm; if the spec-vs-no-spec gap disappears under reset, your effect was accumulation, not spec execution.
4. **Promotion thresholds:** (a) ≥3 *silent* discriminating edges in one FastAPI workspace → real two-arm study; (b) a pydantic v2.x or polars chain yielding ≥2 verified silent introduce-then-use edges → add it for shape-variety and cross-repo generalization; (c) any candidate whose later-change *public* tests still pass after subtract-one-change → that edge is co-location, discard immediately regardless of prose or metadata.
