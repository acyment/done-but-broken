# Round Three: Substrate Selection for the Silent-Regression Agent Experiment

**Bottom line:** the full conjunction (a shipped, culprit-named *silent* regression + a plausible establishing item that pins the trap's precondition + a natural acceptance surface, all in one repo and window) is **rare**, and this session found exactly **one strong candidate per shape** — Immich for the service-shaped arm and TimescaleDB for the library-shaped arm — not a slate. Build the two-substrate study on those two, but go in knowing the library-shaped trap (TimescaleDB) is fossil-grade while its establishing item is weak, and the service-shaped candidate (Immich) has the strongest surface but slightly softer trap certification and higher memorization risk.

## 1. CRITIQUE FIRST

Round one selected for loud edges; round two under-weighted whether anyone would actually have scenario coverage where the trap hits. This round's hole is in the establishing-item requirement itself, and it is a two-sided failure: the criterion is simultaneously **gameable** and **systematically unsatisfiable**, and the same candidate can slide between the two. It is gameable because almost any observable behavior can be retrofitted into a plausible-sounding Given/When/Then after the fact — "surely someone would have written a scenario asserting the birthday reads back correctly" — so a scout can manufacture an establishing item for nearly any trap by describing the happy path at a comfortable altitude. It is unsatisfiable, honestly, because silent-wrongness fossils almost always live in a **narrow edge configuration** (a photo with no EXIF timezone on a server ahead of UTC; an `int2` column under a *composite* bloom filter queried with a cross-type literal), and real acceptance suites are written against the happy path and the common configuration, not the precise precondition the regression exploits. The tell that separates a real establishing item from a retrofitted one is whether a scenario the team plausibly wrote **pins the precondition** the trap needs — a non-UTC server timezone, the specific column type — rather than merely "covering the feature." If the establishing scenario only exercises the default configuration, the accumulated suite never fires no matter how faithfully it is written, and the candidate fails silently in exactly the way the experiment is trying to detect in the agent.

The second hole is an interaction the round's own ranking creates. The round (correctly) rewards service-shaped, intent-level surfaces over internal plumbing — but the shipped-silent-wrongness fossils concentrate in exactly that plumbing: hash-function cross-type compatibility, timezone/calendar arithmetic, serialization boundaries. The more service-shaped and intent-level the acceptance scenario, the **coarser** it is, and the more likely a narrow edge slips through it untested. So criterion 4 (natural surface) and criteria 1–2 (a catchable silent trap) are in tension: pushing up the naturalness of the surface tends to push down the probability that the establishing scenario pins the trap's precondition. The requirement should therefore be strengthened to demand that the establishing scenario name the precondition, and each candidate below is scored on whether it can meet that stronger bar — not the weaker "would someone have tested this feature at all."

## 2. NOMINATIONS

### Nomination A — Immich (service-shaped: self-hosted photo/video server, HTTP API + OpenAPI contract)

Immich is a Dockerized service (`docker compose up`, server on port 2283) with a documented REST API and an OpenAPI spec, plus a metadata/date subsystem that is a proven, repeated source of silent date/timezone wrongness. This is the strongest service-shaped surface I found: date and timezone behavior is directly observable over the HTTP API (asset timeline ordering, person birth dates), and Given/When/Then over that API is exactly how such behavior is specified in industry.

**Trap item (fossil-certified, post-January-2026).** The date-only serialization bug fixed in **PR #28810** ("fix(server): respect timezone in iso date string encoding"). Confirmed on the PR: jrasm91 merged commit `e94e22f` into `main` on June 3, 2026 (53 checks passed), closing user-reported issue **#28091** ("Birth dates keep being changed to one day earlier," reporter running a non-UTC server). The tempting implementation is the idiomatic `date.toISOString().split('T')[0]` — in the PR author's own words: "`toISOString().split('T')[0]` shifts the timestamp to UTC, which can roll the calendar date back by one day on servers ahead of UTC." The fix replaces it with local-calendar-field encoding and **adds a regression test**, verbatim from `server/src/utils/date.spec.ts` in PR #28810:

```js
it('should return the local calendar date, not the UTC date', () => {
  const date = new Date(2000, 0, 15); // 15 Jan 2000, local midnight
  expect(asDateString(date)).toBe('2000-01-15');
});
```

This is silent wrongness: the code compiles, the ordinary suite passed (it ran in UTC, where the bug is invisible), and the API returns a birth date one day early. Certification is strong: user-discovered breakage plus a merged fix that names the defective line and codifies the correct behavior in an added test.

**Forced-touch property.** The defective conversion lives in a **shared helper** (`server/src/utils/date.ts`, `asDateString`/`asBirthDateString`), used by every date-only field the API serializes. A backlog task that adds or reworks any date-only API field cannot be completed without calling or editing that shared helper — the agent is forced into the exact code carrying the earlier behavior, rather than wandering in.

**Establishing item.** The **person birth-date / age feature** in the People API (evidence: issue **#3785**, "New age feature can set birth date in future," which establishes that `birthDate` is a first-class, user-set, API-returned field). Its acceptance scenarios, written at the time, would assert: *Given a person with birthDate 2000-01-15, When I PATCH then GET the person over the API, Then birthDate is returned as "2000-01-15."* That scenario covers the trap's blast radius **only if the test server's timezone is set ahead of UTC** — precisely the precondition the honest establishing item must pin, and the reason this candidate's weakest link is the establishing check (below).

**Second fossil in the same subsystem (reinforces realism, weaker certification).** The no-EXIF-timezone regression: issue **#24107** ("Regression in metadata parsing of files without timezone, UTC assumed instead of local time"), caused by commit `0df70365d725e128530fac8a09f092e72b7f54a7` shipped in **PR #23561** ("feat: exif medium tests," merged Nov 4, 2025), which shipped in release **v2.3.0** (~Nov 12, 2025) — a test-refactor PR that silently changed timezone-fallback from server-local to hardcoded UTC (`timeZone ?? 'UTC'`), reordering timelines. Memorization risk: this culprit is November 2025 (pre-2026); I flag it as older, and its merged-fix status could not be confirmed (community proposal #24116 appears unmerged), so I treat its certification as "user-discovered breakage" (second-best), not fossil-grade.

**Filler items (same subsystem, 2026 window, independent).** `fix: utc time zone upserts` **PR #26258** (merged Feb 2026); `render storage template date/time tokens in UTC (#24350)` **PR #26917**; `fix: null local date time in timeline queries` **PR #26133**; `fix(mobile): keep timezone when editing asset date time` **PR #28978**; `fix(web): preserve timezone when changing timestamp (Closes #25354)` **PR #27095**.

**Surface naturalness.** Very high. Timeline ordering and person birth dates are user-meaningful operations exposed over a documented HTTP API; acceptance scenarios over that API are idiomatic and widely written in industry.

**Least-confident check.** The establishing-item check. A birthDate acceptance scenario written at the time would very plausibly exist, but whether it **pins a non-UTC server timezone** — the precondition without which the trap cannot fire — is uncertain; most test harnesses default to UTC, in which case the accumulated suite would not catch the trap. Semantic knockout and loudness are easy to verify; scenario-precondition coverage is the risk.

### Nomination B — TimescaleDB (library/SQL-shaped: Postgres extension, SQL is the public contract)

**Trap item (fossil-certified, post-January-2026).** Issue **#9579** ("bloom filter false negatives for int2 columns," opened by maintainer akuzm on Apr 13, 2026), fixed by **PR #9581**. The reproduction in the issue is exact and damning — verbatim from the issue body: `SELECT count(*) FROM t WHERE si = 1::smallint;  -- 50 (correct)` versus `SELECT count(*) FROM t WHERE si = 1;  -- 0  (WRONG)` (an int4 literal against the int2 column silently drops all matching rows). The maintainer's own diagnosis, verbatim: "The hash we use for int2 is not crosstype-compatible, as is required for all the int hashes. Apparently I forgot about this case because a bloom filter on int2 doesn't make sense storage-wise, but this becomes a problem with composite bloom filters." This is textbook silent wrongness — compiles, ordinary suite passes, query returns wrong (missing) rows. And it shipped: composite bloom filters were released in **v2.26.0 (2026-03-24)** via **PR #9372 / #9374**; per the timescale/timescaledb release notes the defect shipped in v2.26.0, was reported in issue #9579 on Apr 13, 2026, and was fixed only in **v2.27.0 (2026-05-12)** — **exactly seven weeks** of silently wrong `SELECT` results in a shipped release, with the v2.27 notes confirming affected databases must have the bad indexes dropped before upgrade.

**Forced-touch property.** Composite/multi-column bloom pushdown is implemented in the same sparse-index hashing path that carries single-column bloom-filter correctness. A task to extend bloom pruning to a new operation or column combination (the actual 2.26/2.27 work) forces editing the hash/filter-build code — the tempting path reuses the existing per-type hash, which for int2 is not cross-type safe. The trap cannot be completed without touching the code the earlier "bloom filters never drop matching rows" behavior lives in.

**Establishing item.** The original **bloom filter sparse index feature, PR #7638** ("Implement bloom filter sparse index for compressed tables," shipped v2.20.0, 2025-05-15). Its acceptance scenarios, expressed as SQL result-set assertions (pg_regress-style, the idiomatic contract test for a database), would assert: *After compressing a chunk, a `WHERE col = value` query returns exactly the rows it returned before compression* — i.e., bloom pruning never yields a false negative. That contract, extended to the int2/composite case, is exactly what the trap breaks.

**Filler items (same subsystem, 2.26/2.27 columnstore window, independent).** `Use bloom filters to reduce decompression during UPDATE/DELETE` **PR #9399**; `Use bloom filters to eliminate decompression ... during UPSERTs` **PR #9374**; `Enable vectorized aggregation ... time_bucket() filters` **PR #9443**; `Push down scalar array operations into the columnar metadata scan` **PR #9192**; `Fix segfault in bloom1_contains` **PR #9417**; `Drop existing sparse indexes when dropping columns` **PR #9583**.

**Surface naturalness.** Moderate. SQL query correctness *is* the product's public contract and pg_regress result-set assertions are the idiomatic acceptance test for a database — so this is a docs-level contract, not raw internal plumbing. But it is closer to library plumbing than Nomination A's HTTP API, and a lay engineering leader reads it as "database internals" rather than "a service backlog."

**Least-confident check.** The establishing-item plausibility check under the stronger bar. General query-correctness scenarios exist in abundance, but whether a scenario written for the bloom feature would specifically exercise a **composite bloom filter on an `int2` column queried with a cross-type (`int4`) literal** is doubtful — that is precisely the narrow edge the maintainer admits he overlooked. Trap certification is essentially unimpeachable (fossil-grade); the risk is that the accumulated suite would plausibly not have covered the edge.

## 4. SEARCH STRATEGY

**Queries that worked.** `merged PR "regression" "introduced in" fixed shipped release 2026 CLI`; `bug fix regression shipped wrong output timezone pagination 2026 github` (surfaced both TimescaleDB #9579 and the Immich timezone megathread in one shot); `timescaledb bloom filter compressed int2 wrong results PR 9579`; `immich exif timezone regression commit 0df7036 PR 24116`; `immich issue 28091 birth date one day earlier UTC bug`; `timescaledb 2.27 composite bloom filter PR 9374 SELECT UPSERT feature`. The highest-yield pattern was pairing a **symptom of silent wrongness** ("wrong output," "incorrect result," "assumed UTC," "false negatives") with a year and a subsystem noun, rather than the word "regression" alone.

**GitHub search qualifiers to continue by hand.** In GitHub PR search: `is:pr is:merged "regressed by" merged:>2026-01-01`, `is:pr is:merged "introduced in" in:body label:bug`, and repo-scoped `repo:immich-app/immich is:pr "timezone" in:title is:merged`, `repo:timescale/timescaledb is:issue "incorrect result" OR "false negative"`. In issue search, `label:"changelog:bugfix" "utc"` inside immich, and `label:bug "wrong" "compress"` inside timescaledb, both return clusters. The release-notes route (reading a repo's Releases page and grepping the "Backward-Incompatible Changes"/"Bugfixes" sections for "could lead to ... wrong/incorrect") was the single most efficient way to find *shipped* silent regressions with culprit PR numbers.

**Dead ends.** The GitHub merge-queue April-2026 incident and the Claude-Code cache-TTL regression are infrastructure/service-side incidents, not code-in-a-repo tasks — no local build, excluded. Generic CLI changelogs (pip-tools, fonttools) had regressions but they were loud (crashes) or trivially caught. Forgejo/Gitea API fixes were mostly loud errors (500/404) or migration-only, not silent wrong answers over a stable API.

**Vein not exhausted.** The **Apache Solr/Lucene** `Changes.html` (e.g., SOLR-17649 "regression of faceting on multi-valued EnumFieldType," SOLR-17221 case-only query-param merge in distributed search) is a rich, service-shaped (HTTP query API) vein with many "regression of X introduced by Y" entries I did not fully mine; and **TimescaleDB v2.27.2**'s noted "correctness bug in grouped aggregation on the columnstore" is a second in-repo fossil I did not open. Both are worth an hour by hand.

## 5. SUPPLY, RETARGETED

The scarce joint property is now **trap (shipped silent wrongness, culprit identified) + a plausible establishing item that pins the precondition + a natural acceptance surface, all in one repo/window.** Honest denominator from this session: I inspected roughly a dozen candidate repos/threads reached through ~15 searches; of those, exactly **two** repositories cleared the full conjunction (Immich, TimescaleDB), a small handful had the trap but not a service-natural surface or not a confirmed shipped-and-fixed fossil (Solr — promising but unmined; Gitea/Forgejo — mostly loud; pip-tools/fonttools — loud or trivial), and the remainder failed at the first gate (loud failures, or infrastructure incidents with no local build).

**Where it concentrates.** Silent-wrongness-with-named-culprit concentrates heavily in **date/timezone/calendar code** (Immich is a whole subsystem of it) and in **query-engine/compression correctness** (TimescaleDB columnstore). Those two domains are where "compiles, tests pass, wrong answer, professional took the tempting path" recurs. Service-shaped instances cluster in the timezone/serialization family; library-shaped instances cluster in the storage/query-engine family.

**Is supply rich enough for a two-substrate study (one service-shaped, one library-shaped)?** Yes — but only just, and the two substrates are unbalanced in the way this round cares about. The library-shaped slot (TimescaleDB) has a **fossil-grade** trap but a weaker establishing item; the service-shaped slot (Immich) has a **strong, natural** surface and establishing item but slightly softer trap certification and higher memorization risk. That is enough to run the intended two-substrate design, and the two are genuinely different shapes. It is **not** rich enough to claim the conjunction is common: the honest read is that the full conjunction is **rare**, and I found precisely one strong instance per shape rather than a slate to choose from.

## 6. WALK-AWAY

**Substrate class I would refuse even if it scored well:** external-service/platform incidents (the GitHub merge-queue corruption, the Claude-Code cache-TTL regression). They are textbook silent wrongness with clean fossils, but there is no local repository to browse, no local build, and no way to reconstruct before/after state or run the ordinary suite — they violate the hard exclusions outright, and no scorecard number should rescue them.

**If the full conjunction cannot be found at this budget:** it *can* be found, but thinly — the honest position is that the conjunction is rare and I have one credible candidate per shape, not a comfortable margin. If forced to name the single strongest target that exists, it is **Immich's date/timezone subsystem** for the service-shaped arm (natural HTTP surface, post-January-2026 fossil in PR #28810 with an added regression test, trivial Docker reproduction) paired with **TimescaleDB #9579/#9581** for the library-shaped arm (fossil-grade trap). The strongest *nearby* target I did not fully verify — and the one to open first if either primary fails local verification — is **Apache Solr's** faceting/query-parameter regressions (SOLR-17649, SOLR-17221): a service-shaped HTTP query API with a documented contract and a dense "regression introduced by" changelog.

## Recommendations

1. **Commit to the two-substrate pairing now, but verify in this order.** Start local verification with **TimescaleDB #9579** because its trap is fossil-grade and the repro is a five-line SQL script — cheapest, highest-certainty confirmation. Build the extension in Docker at v2.26.0, reproduce `WHERE si = 1` returning 0, then confirm v2.27.0/PR #9581 fixes it. **Benchmark that would change the plan:** if the ordinary pg_regress suite at v2.26.0 already fails on this (i.e., the regression was loud), drop TimescaleDB — but the maintainer's own account says it shipped, which means the suite passed.

2. **For Immich, resolve the establishing-item risk before investing.** Stand up the server via `docker compose up` with `TZ` set **ahead of UTC** (e.g., `Australia/Sydney`), then over the REST API PATCH a person's `birthDate` to `2000-01-15` and GET it back; confirm the pre-#28810 build returns `2000-01-14`. **Threshold:** if you cannot make the trap fire without the non-UTC `TZ` (i.e., it also fails in UTC), the establishing scenario must explicitly pin the server timezone — bake that into the executable-acceptance condition, or the catchable-suite arm is invalid.

3. **Write the establishing scenarios to pin preconditions, per the critique.** For Immich, the birthDate scenario must run under a non-UTC `TZ`; for TimescaleDB, the query-correctness scenario must include an `int2`/`smallint` column under a composite bloom filter queried with a bare integer literal. A happy-path scenario in the default configuration is a decoy that will make both conditions look identical.

4. **Use the fillers listed as the independent backlog items** (Immich: #26258, #26917, #26133, #28978, #27095; TimescaleDB: #9399, #9374, #9443, #9192, #9417, #9583). They share the subsystem and window, need no deep analysis, and read as a genuine backlog rather than a curriculum.

5. **Hold Apache Solr in reserve** as the backup service-shaped arm; mine `Changes.html` for SOLR-17649/SOLR-17221 by hand before the next round if either primary fails local reproduction.

## Caveats

- **Immich secondary fossil (#24107) is pre-2026 and its fix is unconfirmed.** The `0df7036`/PR #23561 timezone-fallback regression shipped Nov 2025; the proposed fix #24116 appears to be a community proposal that was likely never merged, and I could not identify the merged fix PR. Use #24107 only as corroborating evidence of a bug-prone subsystem, not as the trap.
- **The primary Immich trap (#28810) and its establishing item are on different, though adjacent, API surfaces** — the fix is a shared date-serialization helper; the establishing item is the birthDate/People feature. The forced-touch story is strong (shared helper), but the "same feature" framing is looser than TimescaleDB's, where trap and establishing item are literally the same bloom-filter code path.
- **Establishing-item plausibility is inferred, not verified, for both candidates.** I did not locate the actual acceptance/scenario tests that existed at feature-introduction time; I reconstructed what they *would* assert. The verification preview's establishing-item check is precisely where both candidates are weakest, and TimescaleDB (score 2) is weaker than Immich (score 3).
- **Contamination/memorization risk is real for Immich** (108,353 stars; heavily discussed date bugs). The post-January-2026 date of the primary trap mitigates but does not eliminate this; a model may have absorbed the general "toISOString UTC rollback" pattern independent of this repo.
- **All PR/issue/commit references were read from GitHub during this session**; both repositories were browsed (release pages, issue bodies, PR conversations). No sequence here is synthetic or projected. Local build success (Docker) is asserted from documented install paths, not yet independently executed — step 1–2 above exist to confirm it.
