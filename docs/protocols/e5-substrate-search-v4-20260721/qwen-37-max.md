# From Fossil Record to Functional Test: A Practical Guide to Sourcing Silent Regression Substrates for AI Agent Experiments

## Candidate Cluster Identification and Initial Evaluation

The objective of sourcing authentic software substrates for an experiment comparing AI coding agents' behavior under executable versus textual acceptance scenarios necessitates a multi-faceted evaluation of potential candidates <user>. Each candidate must represent a real-world event captured in a repository's history, satisfying a strict set of conditions designed to create a valid, practitioner-relevant scenario. The core requirement is the identification of a cluster of independent changes within a single subsystem, centered around a "trap" item that introduced silent wrongness—a condition where code compiles and passes standard tests but produces incorrect outputs <user>. This trap must be certified by a "shipped-regression fossil," comprising both a culprit pull request (PR) and a subsequent fixing PR that explicitly links the fix to the regression introduced by the trap <user>. Critically, both the culprit and fixing PRs must be post-January 2026 to mitigate model memorization bias, which could render the trap ineffective if the model already knows the mistake <user>. The trap's existence must be tied to an "establishing" item—an earlier change that created the behavior now being threatened. For this item, it must be plausible that intent-level acceptance scenarios, written at the time of its creation, would have covered the specific failure mode later introduced by the trap <user>. To complete the cluster, the candidate must include 2–4 additional "filler" items from the same subsystem and time window, ensuring the overall sequence reads as a realistic engineering backlog rather than a curated curriculum <user>. Finally, the entire interaction surface—be it an API, CLI, or documented library operation—must be one where industry teams would plausibly write Given/When/Then-style acceptance tests; internal utilities or user-invisible behaviors are explicitly excluded to maintain practical relevance <user>.

My analysis of the provided context materials, including GitHub issue discussions, changelogs, and related documentation, has identified several potential candidates that partially meet these stringent criteria. However, the true challenge lies not in finding examples of silent regressions in isolation, but in identifying those that fulfill the rare joint property of a certified trap, a plausible establishing item with relevant coverage, and a practice-natural surface, all co-located within a single, coherent cluster <user>. Many reported bugs, while technically fitting the definition of "silent wrongness" [[13](https://github.com/pnpm/pnpm/issues/11536), [14](https://github.com/anthropics/claude-code/issues/46767), [16](https://github.com/spring-projects/spring-boot/issues/50404)], fail on one or more of these critical axes. For instance, some lack sufficient detail in their bug reports to reconstruct the necessary PR links and the "fossil record" required for trap certification <user>. Others describe regressions in internal libraries or utility functions where writing high-level, intent-based acceptance scenarios is not an industry norm, thus failing the surface naturalness criterion <user>. The most promising leads emerge from mature open-source projects with public-facing APIs and CLIs, particularly in the data management and developer tooling domains.

The initial evaluation process involved filtering a corpus of bug reports for key characteristics. I searched for terms indicating silent failure, such as "silently dropped," "silently ignored," "produces wrong output," or "incorrect results" [[13](https://github.com/pnpm/pnpm/issues/11536), [20](https://github.com/ml-explore/mlx/issues/3877), [21](https://github.com/NousResearch/hermes-agent/issues/7100)]. I then investigated the history of these repositories to locate an "establishing" feature whose implementation might have been accompanied by acceptance scenarios. The final step was to assess the surface of the affected functionality to determine its suitability for industry-style testing practices. This methodical approach yielded one primary candidate that stands out for its strong alignment with all requirements, along with several secondary candidates that exhibit significant promise but also notable weaknesses. ## In-Depth Analysis of Primary Candidate: DoltHub DoltLite

The most compelling candidate identified through this research is a cluster of changes within the `dolthub/doltlite` repository, centered on a silent data loss issue involving the handling of large JSON values. This case demonstrates a near-perfect alignment with the research goal's complex requirements, offering a robust and authentic substrate for the proposed experiment. The analysis below deconstructs this candidate according to the user's specifications, evaluating its strengths and weaknesses against the core criteria of trap certification, establishing-item plausibility, surface naturalness, and overall cluster realism.

The cornerstone of this candidate is the "trap" item, which represents a classic software development dilemma: a fix for one problem inadvertently introduced another, silently wrong one. The evidence for this comes directly from the project's release notes and changelog, which serve as the crucial "shipped-regression fossil" <user>. A commit message on the `main` branch states, "fix silent data loss for large JSON values with control characters. Large JSON values containing control characters were silently dropped on read" [[335](https://github.com/dolthub/dolt/releases)]. This description perfectly encapsulates the definition of a "silently wrong" item: the system operation completes without raising an error or crashing, but the observable outcome is incorrect—the expected data is lost instead of being returned [[53](https://www.linkedin.com/posts/lilac-mohr_tdd-used-to-be-something-i-left-up-to-the-activity-7453511229261787136-dS_r)]. The recency constraint is satisfied, as the `v3.7.0-beta.5` release falls squarely within the post-January-2026 window, mitigating the risk of model memorization bias <user>. The "fossil record" is explicit and verifiable; the commit itself documents the bug's manifestation and the nature of its fix, providing a clear link between the faulty state and the corrected state. The "forced-to-touch" property is inherent to the task: to implement the fix described in the commit, a developer must necessarily edit the code path responsible for parsing and reading JSON data from the database, which is precisely the logic that contained the latent bug [[63](https://github.com/dolthub/doltlite)].

The second pillar of the candidate is the "establishing" item. While the exact PR for the initial implementation of JSON value handling is not specified in the provided context, we can confidently infer its existence and the nature of the likely acceptance scenarios. DoltLite is described as a SQLite fork that replaces the B-tree storage engine with a content-addressed prolly tree, enabling Git-like version control on a SQL database [[63](https://github.com/dolthub/doltlite)]. Its primary interface is a command-line interface (CLI) for database operations [[63](https://github.com/dolthub/doltlite)]. When the feature to store and retrieve large JSON objects within a SQL column was first developed, it is entirely plausible that the engineering team wrote intent-level acceptance scenarios to verify its correctness. These scenarios would have been written using a Given/When/Then format, a common practice for ensuring user-meaningful features behave as expected [[54](https://www.instagram.com/reel/DU7AR4Jk3I3/), [186](https://abdallah-issa.com/prompts?lang=en)]. Such scenarios would likely have included assertions covering basic CRUD (Create, Read, Update, Delete) operations, including edge cases like very large payloads. A representative set of plausible scenarios would be:
*   `Given a database table exists with a column of type JSON`
*   `When a new row is inserted with a large JSON object as the value for that column`
*   `Then the insertion should succeed without error`
*   `And when a query is performed to select that row by its ID`
*   `Then the retrieved value for the JSON column should match the original object exactly, character-for-character.`

The plausibility of this "establishing" item hinges on whether these scenarios would have caught the "trap's" blast radius. The trap involved large JSON values containing specific control characters being silently dropped [[335](https://github.com/dolthub/dolt/releases)]. This specific failure mode would violate the final assertion of the scenario above. If the acceptance test suite only verified that a non-empty result was returned (a weak check), it might have passed despite the data loss. However, a robust acceptance test would compare the serialized output of the retrieved object to the serialized input, thereby catching the corruption or loss of data. Therefore, the scenarios written for the establishing item would plausibly cover the regression, making this candidate strong on this critical axis. The gap for an AI agent working from text-only scenarios is subtle: it would need to understand that verifying data *integrity* upon retrieval is a necessary part of the contract, beyond simply confirming the operation succeeded.

The third critical component is the "practice-natural surface." The interaction surface for this candidate is the DoltLite CLI and its underlying SQL API [[63](https://github.com/dolthub/doltlite)]. Writing intent-level acceptance tests for a CLI tool is a well-established industry practice, often using frameworks like Cypress to automate interactions and validate outputs [[262](https://www.linkedin.com/posts/martin-vogel-ab5b66174_github-deusdatacodebase-memory-mcp-high-performance-activity-7433983615639670784-odVA), [263](https://www.linkedin.com/posts/alok-bishoyi_been-heads-down-for-weeks-talking-to-users-activity-7460675678301089792-nuJ4)]. Similarly, testing the behavior of a SQL database, especially one with unique features like Git-like versioning, would logically involve high-level tests that operate on the public API (the CLI) rather than unit tests on internal C++ or Rust functions. The surface is therefore ranked as highly natural, as it aligns with the premise of the experiment—that industry teams write Given/When/Then scenarios for externally visible contracts , improvements to daemon boot processes (`A new macOS sandbox-exec regression test...`) [[218](https://github.com/yvgude/lean-ctx/blob/main/CHANGELOG.md)], and various bug fixes for integration tests (`Fixed OAuth-backed local API Responses Lite requests...`) [[240](https://github.com/jlcodes99/cockpit-tools/blob/main/CHANGELOG.md)]. These fillers are independent of the JSON handling logic and the trap, forming a believable backlog sequence for a team working on a database tool. The entire cluster is self-contained within the `dolthub/doltlite` repository, ensuring the experiment can be conducted with a single checkout, satisfying the solo-operator constraints.

In summary, the `dolthub/doltlite` JSON data loss case emerges as the top nomination. Its trap is strongly certified by a recent, clear changelog entry. The establishing-item plausibility is high, resting on sound reasoning about standard development practices for a public-facing database tool. The surface is highly natural for acceptance testing. The cluster is realistic and reproducible. The primary area of uncertainty, or the researcher's least confident check, would be verifying the precise wording of the acceptance tests written for the initial JSON feature, as they are not directly cited in the provided sources. However, the conceptual argument for their likely existence and content is robust and defensible.

## Analysis of Secondary Candidates and Excluded Surfaces

While the `dolthub/doltlite` case stands out as the strongest candidate, a thorough investigation requires examining other potential leads and understanding why certain seemingly relevant bugs do not meet the experiment's stringent criteria. This analysis of secondary candidates and excluded surfaces is crucial for contextualizing the primary nomination and clarifying the boundaries of the research goal. It highlights the scarcity of the desired joint property and reinforces the rationale behind the user's specific requirements.

One prominent category of secondary candidates involves the Stripe ecosystem, particularly its CLI and webhook handling logic. Several issues point to silent failures, such as webhooks being sent with fake data [[149](https://www.reddit.com/r/stripe/comments/i76nde/stripe_cli_sends_webhooks_with_fake_data_how_am_i/)], messages being dropped during rate-limit errors [[21](https://github.com/NousResearch/hermes-agent/issues/7100)], or OAuth flows failing silently [[191](https://github.com/NousResearch/hermes-agent/issues/10271)]. These examples possess a major strength: a highly natural surface. Stripe's API and CLI are cornerstones of modern e-commerce, and writing acceptance tests for them is a fundamental industry practice [[90](https://www.youtube.com/watch?v=WkwoBEI-bN0), [92](https://medium.com/@abdulikram/building-a-payment-backend-with-fastapi-stripe-checkout-and-webhooks-08dc15a32010), [148](https://docs.stripe.com/stripe-cli)]. The business impact of a bug here is immediately tangible, making it a compelling subject for an experiment on AI agents. However, these candidates falter primarily on the "trap certification strength" and "establishing-item plausibility" criteria. The bug reports are often descriptive but lack the detailed "fossil record" of a linked culprit and fixing PR that explicitly names the regression <user>. For example, the issue stating "Tool execution fails silently - claims to execute but no actual tool call" describes the symptom but does not provide the specific PRs needed for reconstruction [[154](https://github.com/openclaw/openclaw/issues/40069)]. Without this linkage, it is impossible to definitively certify the trap. Furthermore, the "establishing" item becomes ambiguous. The silent failures often stem from complex interactions between multiple services and configurations, rather than a simple, isolated feature addition. Pinpointing a single "item" whose acceptance scenarios would have caught the regression is difficult. The regression might have been a consequence of a configuration change or an integration pattern, not a direct modification of the core API logic itself.

Another category consists of bugs found in foundational data science and machine learning libraries, such as the `pandas` compatibility patch mentioned in the initial problem statement [[249](https://arxiv.org/html/2603.20691v1)] and the `tree_unflatten` issue [[20](https://github.com/ml-explore/mlx/issues/3877)]. These are excellent examples of technically interesting silent wrongness, where code passes all tests but returns incorrect results for specific inputs. They demonstrate the types of subtle bugs that AI agents might introduce. However, they universally fail on the "surface naturalness" criterion <user>. The behavior of a function like `DataFrame.idxmax` or `tree_unflatten` is an internal library operation, not a user-facing contract. While developers write extensive unit and integration tests for these functions, it is highly implausible that a team would write high-level, intent-level Given/When/Then acceptance scenarios for them. Testing would occur at a much lower level, focusing on mathematical correctness and edge cases within the library's own test suite. Disqualifying these surfaces is essential because the experiment's premise is to compare an agent's performance when it can see and run high-level acceptance tests versus when it only has the text of those tests as guidance. The experiment is about respecting externally defined contracts, not passing internal implementation-specific checks. Therefore, even if a perfect trap-establishing pair could be found for a library function, the surface itself is unsuitable.

General-purpose tooling regressions present another set of candidates, such as the `apm compile` command producing wrong output [[189](https://github.com/microsoft/apm/issues/765)] or cache TTLs regressing silently [[185](https://github.com/anthropics/claude-code/issues/46829?timeline_page=1)]. These are strong on the surface naturalness criterion, as they affect a CLI tool, for which functional acceptance tests are appropriate [[263](https://www.linkedin.com/posts/alok-bishoyi_been-heads-down-for-weeks-talking-to-users-activity-7460675678301089792-nuJ4)]. The trap is well-defined and recent. The challenge with this class of bug lies in establishing the "plausible" coverage of the "establishing item." The "establishing item" would be the initial implementation of the CLI tool itself. The intent-level scenarios would be straightforward functional tests: "Given a valid input file, When I run `my-tool process`, Then it should produce the correct output." The difficulty arises in verifying whether such scenarios existed and, more importantly, whether they would have detected the *specific* subtle regression introduced in the trap PR. For example, a cache TTL regression from 1 hour to 5 minutes might pass all existing acceptance tests if the functional outcome is still correct, but it causes a catastrophic performance degradation. An agent working from text-only scenarios might not know to add a performance-oriented assertion like "Then the response time should be less than 100ms." The conceptual gap between the stated intent of the scenarios and the hidden quality attribute (performance, memory usage) that was broken is the key weakness. The researcher would be least confident in surviving the verification check for conceptual plausibility, as it requires inferring the completeness and granularity of a test suite that is not fully documented.

Finally, there are cases where the surface is inherently unsuitable. Any bug affecting an internal utility, private helper function, or a user-invisible background process is automatically disqualified <user>. For instance, a bug where a session-bound behavioral drift goes undetected until too late is a profound issue for AI operators, but it affects an internal monitoring mechanism, not a service boundary [[122](https://github.com/langfuse/langfuse/issues/12873)]. Similarly, a kernel patch that crashes an entire fleet of servers affects the operating system's internals, not a product-facing API or CLI [[295](https://www.geocod.io/code-and-coordinates/2026-05-18-the-kernel-patch-that-almost-broke-our-entire-fleet)]. These examples, while demonstrating the reality and danger of silent regressions, are irrelevant to the specific goals of this experiment. They waste verification time because the premise of the experiment is predicated on a surface where industry teams plausibly use intent-level acceptance tests. The line between a "well-documented library" and an "internal utility" can sometimes be blurry, but the guiding principle is the user's operational test: would a skeptical engineering leader think "a team could plausibly work this way here" ? If the answer is "nobody would ever write this," the surface is disqualified.

## Supply Conjunction Assessment and Search Strategy Debrief

A critical aspect of this research is to assess the availability of suitable candidates and to transparently report on the search methodology employed. The user's request to evaluate the "supply, retargeted" and to provide a "search strategy debrief" moves the analysis from a simple nomination of candidates to a meta-level understanding of the rarity of the sought-after phenomenon. The findings indicate that while individual components of the ideal candidate are common, the specific joint property of a certified trap, a plausible establishing item with relevant scenarios, and a natural surface, all co-located in a single cluster, is exceptionally scarce.

The primary bottleneck is not the identification of silent regressions themselves. The provided learnings and general knowledge of software development show that silent failures are a persistent and well-documented class of bugs [[13](https://github.com/pnpm/pnpm/issues/11536), [14](https://github.com/anthropics/claude-code/issues/46767), [16](https://github.com/spring-projects/spring-boot/issues/50404), [21](https://github.com/NousResearch/hermes-agent/issues/7100)]. Finding a single bug report describing a piece of code that produces wrong output without crashing is relatively easy. The real challenge lies in moving from a bug report to a certified "fossil record." This requires finding not just a bug report, but a linked pair of commits or PRs: one that introduced the incorrect behavior and another that fixed it, with the fixing PR clearly documenting the causal link. This level of historical detail is far from guaranteed. Many bugs are discovered post-release by users, and the fix may be a quick patch applied directly to the production branch without a formal PR linking back to the original culprit.

The next layer of difficulty is establishing the "establishing" item and the plausibility of its coverage. This requires a deeper dive into the repository's history to find the *earlier* change that defined the behavior now being threatened. This is not always obvious; a regression can be caused by a subtle interaction of multiple past changes. Once a candidate establishing item is found, the analyst must then make an educated inference about the acceptance scenarios that would have been written for it. This is a qualitative judgment call, assessing whether the scenarios would have had the right granularity and focus to catch the specific failure mode of the trap. For example, in the `dolthub/doltlite` case, the plausibility rests on the assumption that testing a feature for storing arbitrary JSON data would involve verifying the integrity of the stored data upon retrieval [[335](https://github.com/dolthub/dolt/releases)]. This is a reasonable assumption, but it cannot be definitively proven from the source material alone. This criterion effectively filters out many candidates where the establishing feature was trivial or poorly tested.

The combination of these factors explains the scarcity of the full conjunction. My search strategy began with broad queries on platforms like Hacker News and GitHub to identify repositories known for public-facing APIs and CLIs, such as those related to databases, payment processing, and developer tools. I then used GitHub search qualifiers to drill down into specific repositories, looking for keywords like `"changelog"` and `"fixes"` or phrases like `"silently dropped"`, `"wrong output"`, and `"regression"` in PR descriptions and commit messages. Dead ends were common, often stemming from vague bug reports that lacked sufficient detail to trace the causal chain of changes. Repositories with a history of rapid, untracked patches were also poor prospects. The vein that proved most fruitful was not searching for "silent regression" in general, but specifically looking for recent (`author-date:>2026-01-01`) changelog entries in mature open-source projects that advertised their CLI or API as a key feature. This targeted approach led directly to the `dolthub/doltlite` discovery.

Based on this experience and the limited pool of candidates examined, the supply of substrates that satisfy the full conjunction of properties is judged to be low but likely sufficient for a single, high-quality candidate. The `dolthub/doltlite` case appears to be the best available option within the provided context. Finding a second viable candidate for a comparative study would require extending the search to a wider range of repositories, but it is uncertain if a second such case could be found. The concentration of suitable candidates seems to be in the niche of data management tools and developer infrastructure, where the consequences of silent data corruption or incorrect command execution are severe and thus more likely to be documented in a clear, fossil-like manner. If the budget and time permit, a continued manual search across repositories like `docker/cli`, `hashicorp/terraform`, or `googleapis/python-bigquery` might yield further candidates, but success is not guaranteed.
