**Design in one line:** a paired, pre-registered A/B on evolving-specification chains forked from the SlopCodeBench harness, where the only manipulated variable is whether the identical OpenSpec Given/When/Then scenarios can execute and push back — with breakage scored on a hidden, growing referee suite that neither arm ever sees.

**Substrate and task structure.** Fork SlopCodeBench (MIT, Docker isolation, native Claude Code and Codex CLI adapters) rather than the paper-1 harness or bespoke repos, because it already implements the two expensive things: agents extending *their own* code across evolving-spec checkpoints, and strict scoring that re-runs all earlier checkpoints' tests. Each chain = seed project + 8–12 sequential requirement deltas. Every delta gets repackaged as an OpenSpec change (markdown delta + scenario blocks in Given/When/Then), byte-identical across arms. If the forked pool yields fewer than ~36 usable chains, author additional chains in the same format before freezing, tag provenance, and pre-register the pooled analysis with provenance as covariate. Each step = a fresh agent session; state is carried only by the repo, including the accumulated `specs/` history. That stateless-sessions-over-a-long-lived-repo pattern is how these CLIs are actually used across days, and it keeps runs comparable; a continuous-session variant is exploratory only.

**The two groups and exactly what differs.** Same repo, same OpenSpec workflow, same accumulated scenario prose. Arm T: scenarios are markdown only; the workflow doc says "before declaring done, review your diff against every accumulated scenario, state pass/fail per scenario, fix what fails." Arm E: identical markdown, plus a frozen step-definition library and a `make verify` target that executes *all accumulated scenarios* — the verified surface grows each step, deliberately stronger than paper 1's original-tests-only gate; the doc says "before declaring done, run make verify; fix failures." Both arms get exactly one structured revise turn after the check/review, so turn count, retry budget, and the self-review ritual are matched; the only difference is whether the pass/fail signal comes from execution or from the agent's own reading. No rollback in either arm: if the step ends failing, the state carries forward and the chain continues — the conservative, practitioner-realistic choice, and a deliberate divergence from paper 1's rollback gate. Step definitions are mechanical translations of the frozen prose, audited line-by-line against the scenario text (Garbervetsky and Fontdevila are natural external auditors) and published, so E cannot smuggle in information the prose lacks. Nothing is prohibited: if a T agent spontaneously mechanizes the scenarios itself, that's the condition being itself — log it, report the contamination rate, ITT primary, per-protocol secondary. Spec and step-def files remain writable for realism; grading uses frozen external copies, and any agent edit that weakens visible checks is logged as a tampering metric.

**What the agent sees each step.** Both arms: current repo; `specs/` with baseline, all archived deltas, and the current delta; identical CLAUDE.md/AGENTS.md except the one verification paragraph above, frozen verbatim; task message "implement change NNN per its delta." Arm E additionally sees, after implementing, the verify output — failing scenario names plus assertion diff — then takes its single revise turn. Arm T instead produces its own scenario-by-scenario verdict list, then takes its single revise turn. Neither arm ever sees the referee or its results.

**Breakage measurement and surface.** Hidden referee = the harness's strict per-checkpoint acceptance tests, accumulated (all tests introduced at checkpoints ≤ k run at k), executed out-of-band after every step in both arms, invisible to agents. Pre-register a coverage map from each visible scenario to referee tests; the referee must include probes no visible scenario covers. A regression event = a referee test that passed at some step j < k and fails at k. Primary endpoint: chain ends with at least one *unflagged* regression — silent breakage shipped — where unflagged means the agent's final report claims completion without disclosing it; flagged-but-unfixed is logged separately. Secondaries: per-step regression hazard (discrete-time mixed logistic: condition, step index, their interaction, chain random intercept); regressions split by covered vs uncovered referee subsets (does executability only guard what it directly checks, or induce generally safer edits?); forward progress on new requirements (the gate's tax — practitioners will ask); tokens and wall-clock per step; tamper rate; T's self-mechanization rate; and false-completion claims, which is the E4 oracle-gap metric carried over unchanged.

**Sample size and power.** Anchors: chain-level regression incidence 40–73% at toy scale, with the frontier multi-step collapse in papers 2–3 arguing for the upper half of that range; a gate-sized effect of 43.2→12.5%. Assume conservatively 50%→20%. Unpaired two-proportion gives 39 chains per arm; paired on chains (McNemar), with discordance plausibly asymmetric (p₁₀ ≈ 0.30–0.35, p₀₁ ≈ 0.03–0.05), gives roughly 27–42 pairs. Freeze at 36 chains × both arms × 2 repetitions on Claude Code as the powered primary — the effectively unmetered subscription — for 144 chain-runs, ~1,500–1,700 agent steps. That holds >80% power for the assumed effect and remains powered for a paper-1-sized relative effect even if the control rate lands at 40%; MDE at 36 pairs is ~25 points off a 50% base. The step-level model (~720 step observations per arm) adds sensitivity beyond the chain-level headline. Codex CLI runs 12–18 chains × both arms × 1 rep inside the metered plan, pre-registered as a directional replication with no power claim. A 4-chain unregistered pilot debugs the harness and checks the base rate; then freeze protocol, verbatim prompts, endpoints, pinned CLI and model builds with auto-update disabled, and analysis code.

**Biggest validity threat and neutralization.** The kill-shot comment is: "the executable arm got answer-key feedback and extra attempts — you measured retry budget, not executability." Three layers against it. First, the turn-matched protocol: both arms get the identical check-then-one-revise structure and the identical instruction to enumerate scenario verdicts, so ritual and retries are equalized and only the signal source differs. Second, the outcome lives on a hidden referee neither arm can see or optimize, so teaching-to-the-test cannot manufacture the primary result. Third, the covered/uncovered split makes the mechanism publicly inspectable: if E wins only on referee tests mirrored by visible scenarios, we say exactly that — narrow guarding — and scope the claim accordingly, pre-registered both ways. Runner-up threat: provider-side model drift mid-experiment; neutralized by pinned builds plus randomized *interleaved* scheduling of arm runs in blocks, so temporal drift is orthogonal to condition. Third: construct drift between prose and mechanized scenarios; neutralized by freezing prose first, literal step definitions, external audit, publication of both. One interpretive guardrail worth pre-registering: TDAD suggests the prose-review ritual itself may be worse than nothing (9.94%), so this contrast estimates executable-vs-prose *under identical ritual*, not vs nothing — budget permitting, a 12-chain requirements-only anchor arm (exploratory) shows hostile readers where both arms sit against no scenarios at all.

**Expected cost profile.** Build: 2–4 weeks part-time for the harness fork, OpenSpec packaging scripts, scenario prose, step-definition library, coverage maps, and telemetry. Runs: the Claude arm's ~1,500 steps at 3–8 minutes each is 75–200 agent-hours — parallel containers throttled to subscription limits, 2–3 calendar weeks; Codex's 24–36 chain-runs fit inside the capped plan; referee and CI compute is local Docker, negligible. Cash ≈ $0 beyond existing subscriptions; the real spend is authoring and audit labor plus calendar time.

## 1. VALIDITY AUDIT

Ranked, worst first. Two of the top three are things your disclosure list missed, and one is disqualifying as written.

**1. Outcome-surface contamination (you missed this; it's the killer).** Your hidden referee *includes the accumulated scenario suite* — the exact artifact Group T watches go red/green every checkpoint and optimizes its retry against. Group T quasi-trains on a large component of the outcome measure; Group C never touches it. Worse, the harness's own checkpoint tests were derived from the same requirements the scenarios were authored from, so even the "independent" referee component is semantically near-duplicate. As written, a positive result reduces to "checked things stay checked," and any competent reader will collapse it to that. Fix: primary endpoint on a treatment-hidden surface only — harness checkpoint tests *plus a deliberately authored held-out probe set per checkpoint* (edge cases, interaction probes, property/metamorphic tests no visible scenario covers), with a pre-registered scenario→referee coverage map and covered/uncovered results reported separately. The scenario suite's own pass rate becomes a manipulation check, not evidence.

**2. Treatment bundling: you named the retry, you missed the selection.** Your ported gate isn't just "feedback + extra attempt" — the *keep-only-if-≥-as-many-checks* rule is best-of-two selection on a surface correlated with the outcome. Selection alone improves hidden-referee results even if the agent learns nothing from the feedback. So the confound in (i) is three-way: information, attempts, selection. Neither of your two options ("matched retry without info" vs "accept asymmetry") resolves it alone. The resolution is a decomposition you get almost free: **grade attempt 1 in Group T on the hidden referee even when it's replaced.** Then you report three curves: C, T-attempt-1 (deterrence — knowing checks will run), T-final (deterrence + feedback + selection). Combined with a matched revise turn in C (self-review, keep the version it judges better), the "extra compute" confound dies and the mechanism split is quantified instead of argued.

**3. Self-verification policy (your ii): allow it, and here's why the alternative is worse.** A ban is unenforceable — where's the line between "running the app" and "testing"? An unenforceable rule gives you unmeasured contamination; an allowed behavior gives you measured contamination. Allow in both groups, log mechanization events (test files authored, ad-hoc assertions run), ITT primary, per-protocol secondary. The ITT claim — *provided* executable scenarios vs frontier-agent initiative — is exactly the product question your thesis needs answered, and it's the only version that survives hostile comment #3 below. Dilution math: if C spontaneously mechanizes in fraction *m* of chains at ~70% effectiveness, a 30-point effect shrinks to ~30·(1−0.7m); at m=0.3 you keep ~24 points and stay powered at the N below; at m≥0.5 you don't — so the pilot must estimate *m* and your pre-registered N formula must condition on it.

**4. Instruction/salience asymmetry, and the TDAD trap (deeper version of your iii).** As written, C's scenarios are ambient text; T's workflow actively surfaces them. That's a salience confound. But the obvious fix — give C a matched "verify all accumulated scenarios hold before declaring done" instruction — walks into TDAD: instructed review without execution measured *worse than nothing* (9.94% vs 6.08%). Whichever C you choose, you're picking a baseline, not avoiding the choice. Recommendation: matched spec-honoring instruction in both (it's the realistic OpenSpec control — the workflow itself says specs are truth), pre-register the TDAD caveat, and if budget allows add a ~12-chain no-scenario anchor arm so readers see where both groups sit against nothing.

**5. Visibility must be structural, not disabled.** If C's repo contains step definitions and a `make verify` target that's merely "off," agents will grep, find it, and run it — contamination plus weirdness. C's repo must contain *only* markdown; the execution infrastructure must not exist. This is also the ecologically honest counterfactual: a team that never invested in mechanization, not one that hides it.

**6. Tampering.** Your gate presumably executes the in-repo suite; the suite is writable; a T agent that edits a failing step definition games the acceptance rule. Execute a frozen external copy (or hash-verify), and log divergence as a tamper metric.

**7. Missing entirely from your plan:** N and a power argument, frozen analysis model, pinned CLI/model builds with auto-update disabled, randomized *interleaved* scheduling of arms (provider-side drift otherwise aliases with condition), repetition count, flake policy (run referee 3× at baseline, quarantine flakes — flaky tests manufacture regression events), token/wall-clock accounting, and a precise event definition (referee test passed at step j<k, fails at k; denominator = promises whose feature ever passed *in this run* — you can't regress what never worked). Absent a frozen N, "you peeked and stopped when it looked good" is unanswerable.

**8. Control-only pilot leaves half the machine untested.** The gate runner, acceptance rule, snapshotting, and step-def plumbing are the novel code; you'll debug them mid-confirmatory-run. Pilot both arms on ~4 chains, exclude from analysis. Keep your escalation ladder, but numeric and capped: e.g., control chain-level regression incidence <25% at 8 checkpoints → extend to 14; <25% again → declare floor and go to Stage 2. Never condition escalation on T results.

## 2. COMPARABILITY VS REALISM

The "ported gate" claim is already false in the parameter that matters most: 2607.01855's gate only ever executes the *original turn's* tests — a frozen surface — while yours executes the accumulated suite, a surface growing ~linearly to 8–12× by chain end. Yours is a substantially stronger treatment, on frontier agents, in agentic mode, on harder tasks (best strict checkpoint rate 14.8% in SlopCodeBench vs toy Python). Zero basis for magnitude transfer; 43.2→12.5 is usable only as a directional prior for power. And note the irony: the one element you kept for fidelity — the acceptance/revert rule — is the element causing the bundling problem in Audit #2, while everything that made the comparison meaningful changed. Verdict: drop "continuation." Keep one lineage paragraph (their gate-vs-restatement result motivates the contrast), publish an explicit delta table, make no cross-paper effect-size comparisons, and justify the acceptance rule on its own terms as a realistic auto-revert-on-red policy — or drop it and go turn-matched.

## 4. FORK VS PLUG

Extract these while reading, ranked by decision weight:

1. **Grader standalone-ness.** Is strict accumulated scoring a self-contained artifact (repo snapshot + checkpoint index → verdicts, own container) or woven through orchestration? This outweighs everything: the grader is your outcome function, and porting an entangled grader is where silent bugs enter.
2. **State-injection surface.** Can the harness run per-checkpoint fresh sessions with repo-carried state and arbitrary workspace files (OpenSpec dir, make target, CLAUDE.md/AGENTS.md control) without patching agent adapters? Byte-identical instruction control across groups lives or dies here.
3. **Per-checkpoint snapshotting.** Needed for attempt-1 grading, tamper diffs, and the passed-at-j-fails-at-k event definition. If SlopCodeBench lacks it, forking loses most of its edge; check whether your harness has it.
4. **Determinism/flake posture.** Pinned images, network isolation, seeds; empirically re-run one chain's grader 3× and record divergence.
5. **Subscription-mode orchestration.** Parallel containers with backoff/session-resume under consumer seat limits (your Claude plan). Calendar time dies here, not in code.
6. **Telemetry.** Full transcripts, tool calls, token counts per step — required for the tax metric and for detecting C-side mechanization.
7. **Task/harness decoupling.** Are chains declarative data cleanly importable, with checkpoint-dependency metadata (which later features presuppose earlier ones — needed for the regression denominator)?
8. **Upstream health.** Open issues against the strict scorer; known grading bugs are disqualifying either way but change which path inherits them.
9. **Your own harness's audit machinery, honestly assessed.** If the pre-registration hooks and frozen-config hashing are aspirational rather than working, the "import" option loses its main selling point.

Decision rule: grader standalone AND upstream lacks snapshotting → import into your harness. Grader entangled OR snapshotting exists upstream → fork. Seriously consider the hybrid — fork the grader as a service, keep your orchestrator — which dominates both if (1) comes back "standalone."
