## 1. TL;DR

**Recommended base: `hit-sdd-bench`** (the TypeScript/bun record repo, GitHub `acyment/done-but-broken`).

- **Confidence: high** versus `hit-sdd-bench-e2`; **medium** versus Option C (fresh repo) — Option C becomes attractive only if you decide the record repo's guardrails should stay frozen (see §6, §8-Q2).
- **Reversibility: good through Phase 0–1.** Everything the brief demands is net-new code in either repo, so the E2 modules could be lifted into a fresh repo later at modest cost. The choice hardens once E2 starts sharing core code with E1 (brief §12 Q7).

The single most decision-relevant finding: **neither repo contains any of the brief's E2.** No drift meter, no procedural substrate, no arms 0/M/H, no agent-maintained-spec inversion, no brief vocabulary — anywhere, in any branch, in either repo. The `-e2` suffix is a red herring: that repo's "E2" is a *different, already-published experiment* (brownfield acceptance-feedback ablation on SWE-bench Live). So the base choice is not "who has more E2 progress" (nobody does); it is "which repo satisfies the brief's standing constraints" — and the hard constraint *E1 must remain runnable* is satisfiable only by `hit-sdd-bench`.

## 2. What each repo is (provenance narrative)

The two repos share **no git history** — different root commits, different remotes, different languages. This is a deliberate two-repo architecture, not a fork:

- **`hit-sdd-bench`** = GitHub **`acyment/done-but-broken`**. TypeScript/bun. 177 commits, single branch `main` (6 commits ahead of origin, unpushed). It is the **scientific-record + E1-harness repo**: the sealed E1 experiment (CartCalc/billing/dispatch sealed tasks, L0/L1 turn protocol, sealed constants v1.0.0), all protocol docs, run cards, evidence-status docs, the proto-paper, and the AGENTS.md discipline rules. Dirty state is docs-only (direction-update prose + 6 untracked protocol docs, including `REDESIGN-PROMPT.md` — a scratch prompt for an **E3** redesign, dated 2026-07-05).
- **`hit-sdd-bench-e2`** = GitHub **`acyment/done-but-broken-harness`**. Python/uv/Docker. 100 commits on one *linear* line (`authored-spec-glm-authoring` ⊂ `main` ⊂ `stage0-executable-spec-probe`; the current branch is 29 commits ahead of `main` and **has no remote counterpart — unpushed**). Its own pyproject says it plainly: *"Code only; the scientific record lives in hit-sdd-bench."* It is the **execution harness for the repo-E2 brownfield program**: SWE-bench Live substrate + Docker oracle, contamination/flake gates, control-vs-treatment runner, the `authored_spec/` study, and a large body of **uncommitted** E3 work (`src/hit_sdd_e2/e3/`, `scripts/`, 16 files) that the operator's 2026-07-05 `REDESIGN-PROMPT.md` treats as the live harness for the *active E3 program*.
- **`hit-sdd-bench.old`** (context only): an abandoned pre-history working copy — a git repo with **zero commits**, containing an earlier incarnation of the project (`private_oracle/`, JS `src/`). Not a candidate; useful only as evidence that the project has already survived one clean restart.

**Critical naming clarification (affects how you read the brief):** the brief's "E1" conflates two repo programs. The brief's E1 *component list* (L1 parser, CartCalc, versioned constants) belongs to **repo-E1** (the TS harness). The brief's E1 *headline result* (~79%→13% self-verification gap) belongs to **repo-E2** (the brownfield ablation, executed by the Python harness, recorded in the TS repo: pooled control gap_rate 0.7865 → treatment 0.1333, family-wise p≈3.4e-10, plus the Qwen 50%→0% replication). The brief's "E2" is a third thing with no lineage in either repo.

## 5. Port list and discard list (given `hit-sdd-bench` as base)

**Mandatory ports from `-e2`: none.** That is the striking result of the audit — brief-E2 v1 (typed schema IR, change-op action space, NL renderer, programmatic HTTP test-gen, conventions layer, drift meter, 3-arm sequential runner) exists nowhere and gets built fresh in the base.

**Optional / deferred ports (each gated on a Phase-0 ADR):**

1. Provider-routing pattern (`_cli/routes.py`: frozen per-model routes) — if the §12 Q1 language ADR picks Python for the harness, or if TS, replicate the *freeze-the-route* idea in the existing preset system.
2. OpenHands / container-tool agent scaffolds — only if E2 arms are driven by real coding agents in containers rather than the E1-style turn protocol. (The brief's black-box HTTP boundary makes the E1-style loop plausibly sufficient for v1.)
3. Snapshot/resume mechanics from `e3/agent.py` (persistent workspace across steps) — as a *design reference* for §12 Q5, not code.

**Discard (i.e., do not carry into E2; all stay untouched in `-e2` where they serve E3/the record):** SWE-bench Live substrate + oracle + sanitizer, contamination and flake gates, control/treatment arm code, the entire `authored_spec/` pipeline, `e3/` itself.

**If instead `-e2` were the base (for completeness):** you would need to (a) port or permanently sibling-link E1 (it's TS — a port is a rewrite, so the two-repo split survives anyway, defeating the point), (b) resolve the fact that "E2" *inside that repo* already names the published brownfield experiment (manifests, NOTICE, run records all use `e2-*` schemas), (c) remove hardcoded absolute paths into the sibling (`_cli/env.py`, `e3/agent.py`, `authored_spec/validate.py`, several examples/scripts), and (d) build all the same net-new machinery while dodging the live uncommitted E3 work. Strictly more work for no inherited benefit.

## 6. Challenges to the brief

1. **The name "E2" is taken.** Brief §13 insists on "HIT-SDD Bench E2 (not a new benchmark name)", but both repos already use E2 to mean the *published* brownfield acceptance ablation (the brief's own headline result!), and E3 is the active program. Reusing "E2" for drift velocity will corrupt the record's naming discipline (AGENTS.md pooling bans are keyed on experiment boundaries). **Recommend the new program get a fresh index (e.g. E4) or a distinct slug ("E-drift"), while keeping the public-facing framing the brief wants.** This is a challenge to the brief's vocabulary section, not to its design.
2. **Brief §1 factual claims are partially falsified** (the brief itself asks for this flagging — "repo wins on facts"):
   - "L2 orchestrator" does not exist; orchestration lives in `e1-harness.ts` / `e1-package-runner.ts` / `runner.ts`, and the real layering is L0/L1.
   - The L1 parser is a strict block-grammar parser whose completion claim is a structured `done_literal` token — not NL claim extraction. §7's "reuse/extend the E1 completion-claim parser" is directionally fine but the 79→13 gap was actually computed by the *Python* harness, not this parser.
   - Constants: sealed lineage is at **1.0.0** (`e1-frontier-sealed-constants-v1.0.json`); "~0.3.3" matches only a superseded pre-seal draft (0.3.4).
   - "Replication of the published results depends on E1 remaining runnable" is imprecise: repo-E1 is fully reproducible from `hit-sdd-bench`, but the **79%→13% headline is not reproducible from either repo alone** — it needs the Python harness at the recorded SHA, and the two headline runs are themselves flagged **non-replay-valid** (patch text/full traces not retained; harness gap since fixed). The preservation requirement should read: *keep repo-E1 runnable in the base AND keep `-e2` frozen/archived at its recorded SHAs.*
3. **AGENTS.md currently forbids the brief's design.** `hit-sdd-bench`'s guardrails freeze exactly two condition IDs and explicitly ban adding arms or HIT-SDD/BDD/Gherkin condition IDs. The brief's 0/M/H design cannot be built in that repo without a deliberate, recorded amendment establishing a new experiment boundary with its own condition vocabulary. This is resolvable (the guardrails were written to protect E1/E2 evidence, not to prohibit future programs) but must be an explicit Phase-0 act, not a silent edit. If you prefer the record repo's guardrails stay literally untouched, that is the one scenario where **Option C (fresh repo, sibling to both)** beats using `hit-sdd-bench` — at the cost of re-establishing constants/manifest discipline and a third repo to maintain.
4. **`-e2`'s enforcement finding is relevant evidence for Arm H mechanics.** The Claim-B/B1 result (frontier failure mode is done-claims over red/absent self-evidence; enforcement framing) directly informs §12 Q3 (gate enforcement mechanics). Not a conflict — but the brief doesn't cite it, and the ADR for Q3 should.
